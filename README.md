## Часть 1: Dockerfile

В ходе выполнения этого задания был написан "плохой" Docker file, создающий контейнер, внутри которого есть все, чтобы запускать Python-программу. 

### Разберем, что не с ним не так:

1. **FROM ubuntu:latest**
   - **Почему плохо:** Со временем версии меняются. Программа, которую мы будем запускать, со временем может перестать работать в новых версиях. Также, внутри образа ubuntu содержится только ОС. В дальнейшем язык и библиотеки придется доустанавливать. 
   - **Как была исправлена**: `FROM python:3.10-slim`.
   - Теперь вместе с ОС у нас уже есть Python фиксированной версии (что поможет избежать поломок и несовместимостей) и предустановленный pip. 

2. **Много RUN слоев**
   - **Почему плохо:** увиличивается размер образа
   - **Как исправлено:** `RUN apt-get update && apt-get install -y python3 python3-pip`.
   - Файл стал чуть-чуть легче и быстре. 

3. **Установка зависимостей без requirements.txt**
   - **Почему плохо:** Так как мы не указали, какую именно версию flask ставить, будет установлена новейшая из имеющихся, а это может привести к несовместимости. 
   - **Как исправлено:** Создали файл "requirements.txt", в котором в формате "flask==2.3.1" указали версию. Затем копируем txt файл в наш образ и подключаем библиотеку: `RUN pip install -r requirements.txt`.
   - Теперь версия зафиксирована, и не возникнет ошибок при воспроизведении файла на разных устройствах в разное время. Также, все нужные вресии и библиотеки удобнее записать в "requirements.txt" и так как мы указали '-r', то будут установлены все библиотеки из файла. 

4. **Использование ADD вместо COPY**
   - **Почему плохо:** ADD автоматически распаковывает архив и тяняет файл по URL. Нам же необходимо скопировать файлы 
   - **Как исправлено:** используем `COPY`.
   - Файл копируется, не скачивается 

5. **Использование CMD**
   - **Почему плохо:** когда мы делаем "Docker stop" контейнер может не закрыться сразу или остаться “висеть”, потому что сигналы от Docker не доходят до Python.
   - **Как исправлено:** иИспользовать JSON-формат для CMD: `CMD ["python3", "app.py"]` .
   - Теперь Python запускается напрямую, без промежуточной оболочки, контейнер хорошо реагирует на "Docker stop". 

6. **Запуск приложения от имени `root`**
   - **Почему плохо:** если злоумышленник получит доступ к контейнеру, то будет иметь root-доступ внутри контейнера. Также любые баги, которые записывают файлы или изменяют конфигурацию, выполнятся с root-привилегиями, что может повредить важные файлы внутри контейнера.
   - **Как исправлено:** создаем непривилегированного пользователя `useradd -m unprtuser` , изменяем владельца файлов и переключаемся на него: `USER unpruser`.
   - Контейнер остаётся безопасным даже если кто-то попадёт внутрь.

### 2 плохие практики при работе с контейнерами (в рантайме)

1. **Хранение баз данных или логов внутри файловой системы контейнера.**
   - **Почему плохо:** при удалении контейнера удаляется и его файловая система. Если база данных не вынесена в Docker Volumes, все данные будут утеряны.
2. **Внесение конфигурационных изменений через `docker exec`.**
   - **Почему плохо:** редактирование файлов внутри работающего контейнера вручную нарушает принцип Infrastructure as Code. Если контейнер перезапустится, изменения исчезнут. Все настройки должны быть либо в Dockerfile, либо пробрасываться через Volumes/ConfigMaps.

## Часть 2: Docker Compose

Аналогично части 1, было написано 2 файла: `bad-compose.yml`, содержащий 7 плохих практик настройки инфраструктуры, и `good-compose.yaml`, предлагающий исправления.

### 7 плохих практик в Docker Compose:

1. **Использование тега `:latest` для сервисов.**
   - **Почему плохо:** тег `latest` не гарантирует консистентность. В случае обновления сервиса сборка может сломаться.
   - **Как исправлено:** используются конкретные версии образов.
   - **Влияние:** гарантируется детерминированность инфраструктуры.

2. **Отсутствие политики перезапуска.**
   - **Почему плохо:** в случае падения контейнера из-за кратковременной ошибки, он не будет запущен до момента, как это сделает администратор.
   - **Как исправлено:** добавлена директива `restart: unless-stopped`.
   - **Влияние:** контейнер автоматически восстанавливается после сбоев или перезагрузки хост-машины.

3. **Отсутствие ограничений по ресурсам.**
   - **Почему плохо:** неограниченный контейнер может использовать значительный объем оперативной памяти хост-сервера, что приведет к зависанию системы и других сервисов.
   - **Как исправлено:** добавлены лимиты `cpus` и `memory` в блоке `deploy.resources.limits`.
   - **Влияние:** система защищена от тяжелых контейнеров, обеспечивается стабильность хоста.

4. **Запуск сервисов без учета зависимостей.**
   - **Почему плохо:** веб-сервер запускается одновременно с базой данных. Соответственно, так как база не готова принимать подключения, то веб-сервер падает с ошибкой соединения.
   - **Как исправлено:** использован `depends_on` с проверкой `condition: service_healthy` (в связке с `healthcheck` в базе).
   - **Влияние:** исключаются ошибки при старте системы, так как веб-часть ждет готовности БД.

5. **Хардкод секретов в файле `environment`.**
   - **Почему плохо:** docker-файлы часто попадают в системы контроля версий, где чувствительные данные становятся доступны внешним пользователям.
   - **Как исправлено:** использована подстановка переменных окружения: `POSTGRES_PASSWORD=${DB_PASSWORD:-default_secure_pass}`.
   - **Влияние:** повышается безопасность: пароли хранятся в локальном файле `.env`, который не попадает в репозиторий.

6. **Проброс порта внутренней БД на хост-машину (`ports: 5432:5432`).**
   - **Почему плохо:** такой подход делает возможным совершение атак на базу данных с внешних портов.
   - **Как исправлено:** директива `ports` у БД удалена. Сервис доступен только внутри сети Docker.
   - **Влияние:** база данных изолирована от внешних угроз.

7. **Отсутствие томов для данных.**
   - **Почему плохо:** вся информация в БД хранится в файловой системе контейнера. При удалении контейнера все данные пользователей удаляются.
   - **Как исправлено:** добавлен том `volumes: db_data:/var/lib/postgresql/data`.
   - **Влияние:** данные сохраняются на диске хоста и не сбрасываются при перезапуске контейнеров.

### Изоляция контейнеров в рамках Compose-проекта

**Задача:** поднять контейнеры `web` и `db` вместе так, чтобы они не имели сетевого доступа друг к другу.

**Метод достижения изоляции:**
по умолчанию `docker compose` создает одну общую виртуальную сеть для всех сервисов в файле, что позволяет им общаться по именам. Для изоляции был применен подход **"разных сетей"**:

1. В разделе `networks` объявлены две отдельные сети: `frontend_net` и `backend_net`.
2. В настройках сервиса `web` указано подключение только к `frontend_net`.
3. В настройках сервиса `db` указано подключение только к `backend_net`.

**Принцип работы:**
Docker использует сетевые пространства имен. Когда контейнеры подключены к разным изолированным сетям, у них нет общего сетевого шлюза или маршрута для обмена пакетами. Если сервис `web` попытается выполнить `ping db`, DNS-сервер докера не сможет разрешить имя `db` для сети `frontend_net`, так как этот сервис находится в другом сетевом сегменте (`backend_net`). Это полностью исключает горизонтальное взаимодействие между контейнерами на уровне сети.
