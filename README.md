## Часть 1: Dockerfile

В ходе выполнения этого задания был написан "плохой" Docker file, создающий контейнер, внутри которого есть все, чтобы запускать Python-программу. 

### Разберем, что не с ним не так:

1. **FROM ubuntu:latest**
   - **Why is it bad:** Со временем версии меняются. Программа, которую мы будем запускать, со временем может перестать работать в новых версиях. Также, внутри образа ubuntu содержится только ОС. В дальнейшем язык и библиотеки придется доустанавливать. 
   - **How to fix**: `FROM python:3.10-slim`.
   - Теперь вместе с ОС у нас уже есть Python фиксированной версии (что поможет избежать поломок и несовместимостей) и предустановленный pip. 

2. **Много RUN слоев**
   - **Why is it bad:** увиличивается размер образа
   - **How to fix:** `RUN apt-get update && apt-get install -y python3 python3-pip`.
   - Файл стал чуть-чуть легче и быстре. 

3. **Установка зависимостей без requirements.txt**
   - **Why is it bad:** Так как мы не указали, какую именно версию flask ставить, будет установлена новейшая из имеющихся, а это может привести к несовместимости. 
   - **How to fix:** Создаем файл `requirements.txt`, в котором в формате `flask==2.3.1` указываем версию. Затем копируем txt файл в наш образ и подключаем библиотеку: `RUN pip install -r requirements.txt`.
   - Теперь версия зафиксирована, и не возникнет ошибок при воспроизведении файла на разных устройствах в разное время. Также, все нужные вресии и библиотеки удобнее записать в "requirements.txt" и так как мы указали '-r', то будут установлены все библиотеки из файла. 

4. **Использование ADD вместо COPY**
   - **Why is it bad:** ADD автоматически распаковывает архив и тяняет файл по URL. Нам же необходимо скопировать файлы 
   - **How to fix:** используем `COPY`.
   - Файл копируется, не скачивается 

5. **Использование CMD**
   - **Why is it bad:** когда мы делаем `Docker stop` контейнер может не закрыться сразу или остаться “висеть”, потому что сигналы от Docker не доходят до Python.
   - **How to fix:** Используем JSON-формат для CMD: `CMD ["python3", "app.py"]` .
   - Теперь Python запускается напрямую, без промежуточной оболочки, контейнер хорошо реагирует на "Docker stop". 

6. **Запуск приложения от имени `root`**
   - **Why is it bad:** если злоумышленник получит доступ к контейнеру, то будет иметь root-доступ внутри контейнера. Также любые баги, которые записывают файлы или изменяют конфигурацию, выполнятся с root-привилегиями, что может повредить важные файлы внутри контейнера.
   - **How to fix:** Создаем непривилегированного пользователя с домашней папкой /app `RUN useradd -m -d /app appuser` , изменяем владельца файлов и переключаемся на него: `USER appuser`.
   - Контейнер остаётся безопасным даже если кто-то попадёт внутрь.

### Плохие практики при работе с контейнерами

1. **Запуск контейнера с привилегиями root и без ограничения ресурсов**
   Некоторые разработчики запускают контейнер так: `docker run -it my-app`. 
   Это может:
   - Дать злоумышленнику доступ к хосту.
   - Сделать файлы, созданные контейнером, недоступными для хоста.
   - Контейнер может начать использовать всю свободную память, замедляя останостьй хост.
2. **Долгое держание контейнеров без мониторинга**
   Допустим, мы запустили контейнер: `docker run -d my-app`. Он работает, приложение отвечает, но никто его не мониторит.
   Это может:
   - Забивать сисетму, если контейнер течет по памяти или файловым дискриптерам. Несколько таких вполне могут остановить холст.
   - Привести к тому, что кнтейнеры превратяться в "черный ящик" и в случае чего будет очень сложно определить, почему приложение перестало работать.

## Часть 2: Docker Compose

В ходе выполнения этого задания был написан "плохой" Docker compose file.

### Плохие практики в Docker Compose:

1. **Использование образов `latest`**
   - **Why is it bad:** `latest` нестабильная версия, может внезапно измениться и сломать проект.
   - **How to fix:** Фиксировать конкретную версию образа `nginx:1.23.3`.
   - Теперь проект не сломается из-за несовместимости версий. 

2. **Использование стандартного порта**
   - **Why is it bad:** Жёсткая привязка к порту 80 делает проект менее гибким — нельзя запускать несколько таких проектов на одной машине без конфликтов.
   - **How to fix:** Использовать переменные окружения для порта `- "${WEB_PORT:-8080}:80"`.
   - Проект можно запускать на любом доступном порту без редактирования файла.

3. **Пароли в открытом виде**
   - **Why is it bad:** Любой, кто имеет доступ к файлу, видит пароли. Это создаёт высокий риск утечки данных и атак на систему.
   - **How to fix:** Использовать Docker secrets или переменные окружения из отдельного защищённого файла.
   - Теперь пароли не хранятся в коде, а Docker подставляет их из безопасного хранилища.
     
4. **Использование стандартной сети**
   - **Why is it bad:** Все контейнеры автоматически видят друг друга. Если появятся дополнительные сервисы, они смогут обращаться к базе данных без ограничений. Это снижает безопасность.
   - **How to fix:** Создать отдельные пользовательские сети для каждого сервиса или группы сервисов.
   - Теперь web и db находятся в разных сетях и не видят друг друга.
   - 
5. **Отсутствие безопасного управления данными**
   - **Why is it bad:** Данные базы просто монтируются в volume без шифрования, резервного копирования и контроля доступа.
   - **How to fix:** Явно указать имя тома и использовать подходящий драйвер. 
   - Данные теперь хранятся безопаснее, а проект более предсказуем и управляем.

### Реализуем изоляцию контейнеров друг от друга

 1. Создадим отдельные сети для web и db:
```yaml
   networks:
     webnet:
     dbnet:
```
 2. Подключим сервисы только к своей сети:
```yaml
    `web:
  networks:
    - webnet

db:
  networks:
    - 
```
   При изоляции докер использует Bridge-сети. Основные принципы таких сетей: 
   - Каждый сервис подключается к своей отдельной сети.
   - Контейнеры в одной сети могут обращаться друг к другу по имени сервиса.
   - Контейнеры в разных сетях не видят друг друга.
